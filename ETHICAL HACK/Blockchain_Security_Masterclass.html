🔧 Blockchain & Ethereum Fundamentals

What is Blockchain?

A blockchain is a distributed ledger where transactions are grouped into blocks, cryptographically linked together in a chain. Key property: immutability - once data is recorded, it cannot be changed without redoing all subsequent work.

What is Ethereum?

Ethereum extends blockchain concept with smart contracts - programs that run on the blockchain and execute automatically when conditions are met. Unlike Bitcoin (just transactions), Ethereum allows arbitrary computation.

Smart Contracts Explained

Simple Analogy:
A smart contract is like a vending machine:
  • Put in money (send cryptocurrency)
  • Machine executes logic (smart contract code runs)
  • Get product (receive output)
  • No middleman needed (trustless execution)

Ethereum Virtual Machine (EVM)

Smart contracts run on the EVM, a sandboxed virtual machine. Every node in the network runs the same code and reaches the same state. This is how consensus is achieved.

Key Concepts:
Gas: Every operation costs "gas" (computational cost in ETH). This prevents infinite loops and DoS attacks.
State: Contracts store state (variables). Each transaction changes state.
Transactions: Externally-owned accounts (users) initiate transactions. Contract calls are part of transactions.

contract SimpleBank {

        require(balances[msg.sender] >= amount);
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success);
        balances[msg.sender] -= amount;
    }
}

⚠️ Top 15 Critical Smart Contract Vulnerabilities

These are the vulnerabilities that have caused $14+ BILLION in losses. Master these and you can audit contracts worth millions.

1. REENTRANCY ATTACK
CRITICAL
What is it?
Attacker calls a function that sends ETH, then the attacker's fallback function calls the same function again before the first call finishes. This can drain all funds.
Why it matters?
The DAO Hack (2016): $50 MILLION stolen using reentrancy. Led to Ethereum hard fork. This single vulnerability has caused more damage than any other.
vulnerable_code.sol - REENTRANCY VULNERABILITY
contract VulnerableBank {
    mapping(address => uint) balances;
    
    function withdraw(uint amount) public {
        // VULNERABLE: External call before state update
        (bool success, ) = msg.sender.call{
            value: amount
        }("");
        require(success);
        
        // This line executes AFTER external call
        // Attacker can call withdraw() again from fallback!
        balances[msg.sender] -= amount;
    }
    
    // Fallback function - executed when contract receives ETH
    fallback() external payable {
        // Attacker calls withdraw() again here
        // balances haven't been updated yet!
        this.withdraw(amount);
    }
}
exploit_code.sol - HOW ATTACKER EXPLOITS IT
contract Attacker {
    VulnerableBank bank;
    
    constructor(address _bank) {
        bank = VulnerableBank(_bank);
    }
    
    function attack() external payable {
        // Step 1: Send initial deposit to bank
        bank.deposit{value: 1 ether}();
        
        // Step 2: Initiate withdrawal
        // This triggers reentrancy in fallback()
        bank.withdraw(1 ether);
    }
    
    // This gets called when bank sends ETH
    fallback() external payable {
        if (address(bank).balance > 0) {
            // Call withdraw() again while balance hasn't updated
            bank.withdraw(1 ether);
        }
    }
    
    // Result: Attacker withdraws 10x the deposited amount!
}

🚨 Impact:

Complete fund drainage. Attacker can extract all ETH from contract. The DAO lost $50 million in minutes.

✅ Fix #1: Checks-Effects-Interactions Pattern

function withdraw(uint amount) public {
    // 1. CHECKS: Validate preconditions
    require(balances[msg.sender] >= amount);
    
    // 2. EFFECTS: Modify state FIRST
    balances[msg.sender] -= amount;
    
    // 3. INTERACTIONS: Only then external call
    (bool success, ) = msg.sender.call{value: amount}("");
    require(success);
}

✅ Fix #2: Mutex Lock (Reentrancy Guard)

contract SafeBank {
    bool locked;
    
    modifier nonReentrant() {
        require(!locked);
        locked = true;
        _;
        locked = false;
    }
    
    function withdraw(uint amount) public nonReentrant {
        // Even if fallback tries to call withdraw() again,
        // locked = true prevents it!
        balances[msg.sender] -= amount;
        msg.sender.call{value: amount}("");
    }
}
2. INTEGER OVERFLOW / UNDERFLOW
HIGH
What is it?
uint256 can hold 0 to 2^256-1. If you add 1 to max value, it wraps to 0. If you subtract 1 from 0, it becomes 2^256-1. Solidity 0.8+ has built-in checks, but older versions are vulnerable.
Real hack:
BeautyChain (BEC) Token (2018): Attacker exploited integer overflow to create 999 quadrillion fake tokens. The token became worthless overnight.
vulnerable_code.sol - INTEGER OVERFLOW
pragma solidity ^0.7.0;  // Solidity 0.7 - vulnerable!

contract VulnerableToken {
    mapping(address => uint) balances;
    
    function transfer(address to, uint amount) public {
        balances[msg.sender] -= amount;  // Underflow!
        balances[to] += amount;           // Overflow!
    }
}

// Attack: User has 10 tokens, sends 11
// balances[attacker] = 10 - 11
// In uint: 10 - 11 = 2^256 - 1 (HUGE number!)

✅ Fix: Use Solidity 0.8+ (has automatic checks)

pragma solidity ^0.8.0;  // Solidity 0.8+ has overflow protection

// In Solidity 0.8+, overflow/underflow reverts automatically
// No need for SafeMath library!
3. MISSING ACCESS CONTROL
CRITICAL
What is it?
Functions that should be restricted to only the owner or specific roles are public to everyone.
vulnerable_code.sol - MISSING ACCESS CONTROL
contract VulnerableContract {
    address public owner;
    uint public funds;
    
    constructor() {
        owner = msg.sender;
    }
    
    // VULNERABLE: Anyone can call this!
    function withdraw() public {
        // No check that msg.sender == owner
        msg.sender.call{value: funds}("");
        funds = 0;
    }
}

✅ Fix: Add onlyOwner modifier

contract SecureContract {
    address public owner;
    
    modifier onlyOwner() {
        require(msg.sender == owner, "Not owner");
        _;
    }
    
    function withdraw() public onlyOwner {
        // Now only owner can call this
        msg.sender.call{value: funds}("");
    }
}
4. UNCHECKED EXTERNAL CALL RETURN VALUES
HIGH
vulnerable_code.sol - UNCHECKED CALL
contract VulnerablePayment {
    function payWithTransfer(address token) public {
        // VULNERABLE: Ignoring return value!
        token.call(
            abi.encodeWithSignature(
                "transfer(address,uint256)",
                recipient,
                100
            )
        );
        // If transfer fails, code continues anyway!
        balance -= 100;
    }
}

✅ Fix: Check return value

function payWithTransfer(address token) public {
    (bool success, bytes memory data) = token.call(...);
    require(success, "Transfer failed");
    balance -= 100;
}
5. UNSAFE DELEGATECALL
CRITICAL
What is it?
DELEGATECALL executes code from another contract in the current contract's context. If the called contract is malicious, it can modify your state or steal funds.
vulnerable_code.sol - UNSAFE DELEGATECALL
contract Proxy {
    address public owner;
    uint public funds;
    
    fallback() external payable {
        // VULNERABLE: Delegatecall to user-provided address
        (bool success, ) = msg.sender.delegatecall(msg.data);
        require(success);
    }
}

// Attacker provides malicious code that modifies owner/funds

✅ Fix: Use trusted implementation addresses only

Store implementation address securely, never allow user to provide it.

Building an Automated Audit System

Professional auditors use automation to check basic issues, then focus manual audit on complex logic:

automated_auditor.py - Python audit framework
import subprocess
import json
from slither import Slither

class SmartContractAuditor:
    def audit_contract(self, contract_path):
        # Step 1: Run Slither static analysis
        slither = Slither(contract_path)
        
        findings = {
            'critical': [],
            'high': [],
            'medium': [],
            'low': []
        }
        
        # Check each detector in Slither
        for detector in slither.detectors:
            results = detector.run()
            
            for result in results:
                severity = result.get('severity', 'info')
                findings[severity.lower()].append({
                    'type': detector.DESCRIPTION,
                    'location': result.get('source_mapping'),
                    'details': result.get('description')
                })
        
        # Step 2: Run Mythril for deeper analysis
        mythril_findings = self.run_mythril(contract_path)
        findings['critical'].extend(mythril_findings)
        
        # Step 3: Custom checks
        self.check_access_control(slither, findings)
        self.check_reentrancy(slither, findings)
        self.check_gas_issues(slither, findings)
        
        return findings
    
    def check_access_control(self, slither, findings):
        # Check for missing onlyOwner modifiers
        for contract in slither.contracts:
            for func in contract.functions:
                # Flag functions that modify state without access control
                if func.is_writing_to_state and not func.is_internal:
                    if not self.has_access_control(func):
                        findings['high'].append({
                            'type': 'Missing Access Control',
                            'function': func.name,
                            'severity': 'HIGH'
                        })
    
    def check_reentrancy(self, slither, findings):
        # Flag functions with external calls before state updates
        for contract in slither.contracts:
            for func in contract.functions:
                external_calls = func.external_calls_as_expressions
                state_updates = func.state_variables_written
                
                if external_calls and state_updates:
                    # Check order: if external call before state update = reentrancy risk
                    findings['critical'].append({
                        'type': 'Potential Reentrancy',
                        'function': func.name,
                        'fix': 'Use Checks-Effects-Interactions pattern'
                    })

auditor = SmartContractAuditor()
report = auditor.audit_contract("contract.sol")
print(json.dumps(report, indent=2))

💥 Real-World Hacks & Lessons Learned

🚀 The DAO Hack (June 2016) - $50 Million Loss

The Vulnerability: Reentrancy in the withdrawal function. The contract sent ETH before updating balances.

What Happened: Attacker drained 3.6M ETH (worth $50M at the time) in a single recursive transaction.

Lesson: Always use Checks-Effects-Interactions pattern. This single vulnerability pattern has caused more damage than any other in blockchain history.

🔐 Parity Wallet Hack (July 2017) - $30 Million Loss

The Vulnerability: Missing access control + delegatecall abuse. The wallet's initialization function was public (should have been private).

What Happened: Attacker called initWallet() and became the new owner, then transferred all funds.

Lesson: Always restrict initialization functions to internal/private. Review each delegatecall carefully.

📊 BeautyChain Token (2018) - Integer Overflow

The Vulnerability: Integer overflow in token transfer. Attacker sent 999 quadrillion fake tokens to an exchange.

What Happened: Token price dropped 99%+ in minutes. The token never recovered.

Lesson: Use Solidity 0.8+ which has built-in overflow protection. Or use SafeMath library in older versions.

💰 Curve Finance Hack (2023) - $61 Million Loss

The Vulnerability: Complex math error in stablecoin pricing. The mathematical formulas had a subtle bug.

What Happened: Attacker exploited the pricing error to drain liquidity pools.

Lesson: Complex math must be reviewed by cryptography experts. Test with edge cases and fuzzing.
